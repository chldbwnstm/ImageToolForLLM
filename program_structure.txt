 ImageToolForLLM — PROGRAM STRUCTURE (설계 초안 / init)

 최종 갱신: 2026-06-23
 상태: 설계 확정 직후 / 코드 작성 전 (skeleton)
 작성 언어: 한국어(설명) + 영어(코드/경로/식별자)
   ※ README 는 OSS 채택을 위해 영어로 작성하지만, 이 문서는 내부 설계 메모라 한국어로 둠.

--------------------------------------------------------------------------------
 0. 한 줄 요약
--------------------------------------------------------------------------------
 스크린샷을 찍고(영역/지정 윈도우) → 그 위에 "영역 박스 + 라벨/메모"를 달아
 영속 저장하고 → 이미지 + 구조화된 reference 를 Claude Code 가 바로 이해하는
 형식으로 넘겨주는 도구.

 핵심 차별점 = "라벨이 달린 영역 reference 레이어"
   (기존 도구 Invoke / ruslanlap / BensonLaur 는 전부 '캡쳐→경로 전달'까지만,
    영역 분할 + reference 영속화는 빈 영역 → 우리가 채운다)

 형태  = Claude Code 플러그인 (skill + slash command + MCP 서버)
 GUI   = 별도 데스크탑 앱이 아니라 "브라우저 탭"으로 처리 (Electron/Tauri 불필요)


================================================================================
 1. 왜 이 구조인가 (의사결정 로그 — 나중에 흔들리지 않도록 박아둠)
================================================================================

 [Q1] Ctrl+C / Ctrl+V 로 안 되나?
   → 안 됨. 클립보드는 이미지 픽셀만 운반. "1번=Menu 버튼" 같은 구조화 reference
     를 못 실음. 그래서 파일 기반 + skill/plugin/MCP 경로가 필수.

 [Q2] 전체를 skill/plugin 으로 만들 수 있나? (Claude Code 확장으로 흡수)
   → 절반만. Claude Code 확장 표면(skill / slash command / hook / MCP / plugin)은
     전부 텍스트·명령·프로세스 레이어. "박스를 드래그로 그리는 GUI 캔버스"는
     그 안에 절대 못 들어감.
   → 반전: GUI 를 "브라우저 탭"으로 빼면 Electron/Tauri 자체가 불필요해짐.
     브라우저(이미 깔려있음)가 곧 런타임. → 무게/스택 논쟁이 통째로 사라짐.

 [Q3] 데스크탑 앱이면 Electron vs Tauri (무겁다 vs 가볍다)?
   → 위 [Q2] 결론으로 사실상 불필요해짐. 캡쳐는 브라우저가 아니라 MCP 서버
     (네이티브 프로세스)가 하므로 브라우저 샌드박스 제약 안 받음.
   ※ (참고) 만약 정말 올인원 데스크탑 앱으로 확장한다면:
       - "셋 다 최소마찰" 우선 → 슬림 Electron (Chromium 내장, desktopCapturer 가
         Linux Wayland 캡쳐까지 공짜 처리, 단 디스크 ~50MB 바닥)
       - "초경량 6MB" 우선 → Tauri (Rust 학습 + Linux 캡쳐 직접 구현 비용)
     하지만 현재 채택 구조는 둘 다 아님 → "플러그인 + 브라우저 캔버스".

 [Q4] 영역 캡쳐 / 지정 윈도우(핸들) 캡쳐가 이 스택으로 되나?
   → 됨. 캡쳐는 MCP 서버(Node)가 네이티브 라이브러리로 수행하므로 OS 캡쳐
     능력 그대로 사용.
       - 영역 캡쳐: 모니터 전체 캡쳐 → 브라우저 캔버스에서 영역 선택/크롭 (쉬움)
       - 지정 윈도우: 윈도우 목록 열거(제목·앱명·좌표·id) → 선택 → 그 창만 캡쳐
   → 라이브러리: node-screenshots (npm)
       - Monitor.all() / monitor.captureImage()
       - Window.all() / window.captureImage()
       - Win/mac/Linux 프리빌트 바이너리 (사용자 Rust 설치 불필요, npm 만으로 끝)
       - 작은 네이티브 .node 모듈 1개 (무거운 런타임/툴체인 아님)

 [Q5] 전역(global) 핫키가 되나?
   → "내가 지정한 키를 데스크탑 어디서나" 식 진짜 전역 핫키는 플러그인 스택만으론 X.
       - Claude Code hook: Claude 이벤트만 잡음, OS 키 입력 못 잡음
       - 브라우저: 포커스 있을 때만, 시스템 전역 불가
       - MCP 서버: 세션 동안만 사는 프로세스라 깔끔한 전역 데몬 아님
   → 대신 핫키 없이 동등한 트리거 가능 (3.4 트리거 설계 참고).
     진짜 커스텀 전역 핫키가 필요하면 작은 OS 헬퍼(옵션)만 따로 추가.

 [확정 스택]
   - 배포 단위 : Claude Code 플러그인
   - MCP 서버  : Node + TypeScript
   - 캡쳐      : node-screenshots (영역/지정윈도우/윈도우목록)
   - GUI       : 로컬 웹서버 + 브라우저 HTML5 canvas (별도 앱 X)
   - 핸드오프  : *.annotated.png  +  *.regions.json  (+ skill 이 해석법 제공)
   - 라이선스  : 코어 Apache-2.0, CLA 적용, Pro 는 별도 비공개 레포(아직 빈)


================================================================================
 2. 2-레이어 모델 (헷갈리면 설계가 꼬임 — 항상 분리해서 볼 것)
================================================================================

 [레이어 1] 캡쳐 + 영역 주석 (진짜 차별점, 손이 많이 가는 본체)
   - 핫키/명령으로 영역·윈도우 캡쳐
   - 캡쳐 이미지 위에 영역 박스 드래그로 그림
   - 각 박스에 번호 + 라벨 + 메모 입력 (1=Menu, 2=Exit ...)
   - 사이드카 파일로 영속 저장 → 다시 열면 박스 복원 (사용자도 안 까먹음)
   → 구현처: MCP 서버(캡쳐/저장) + 브라우저 canvas(그리기 UI)

 [레이어 2] Claude Code 핸드오프 (skill / MCP / plugin)
   - 넘기는 형식 = 하이브리드:
       (a) *.annotated.png  : 원본 위에 번호 박스를 실제로 구워 넣은 이미지(시각 앵커)
       (b) *.regions.json   : 범례(좌표+라벨+메모). 사람이 편집 가능 + 영속 메모리
   - 왜 박스를 이미지에 굽나: Claude 가 "그림 속 1번" ↔ "범례 1번 라벨"을 시각적으로
     대응시키는 게, raw 좌표를 머릿속으로 찾는 것보다 훨씬 정확하기 때문.
   → 구현처: skill(SKILL.md, 해석법) + slash command(트리거) + plugin(포장)

 3. 시스템 아키텍처
================================================================================

 3.1 전체 흐름 (MVP — 슬래시 명령 트리거)
 --------------------------------------------------------------------------------
   Claude Code 안에서
     │  /capture-region   (또는 /capture-window, /annotate <file>)
     ▼
   MCP 서버 (Node, stdio)  ── 도구 호출 수신
     │  1) node-screenshots 로 화면/윈도우 네이티브 캡쳐
     │  2) 로컬 웹서버 시작 (http://127.0.0.1:PORT)
     │  3) 기본 브라우저로 주석 페이지 오픈 (캡쳐 이미지 로드)
     ▼
   브라우저 canvas (webui)
     │  - 영역 박스 드래그로 그림 / 번호·라벨·메모 입력
     │  - (영역캡쳐 시) 전체 화면에서 관심 영역 크롭
     │  - "전송" 클릭 → canvas 를 PNG 로 export + regions 데이터 생성
     │  - POST /submit  { annotatedPng, originalPng, regions }
     ▼
   MCP 서버
     │  4) 파일 저장:  <out>/<name>.annotated.png  +  <name>.regions.json
     │  5) 웹서버 종료
     │  6) 도구 결과로 Claude 에 파일 경로(+요약) 반환
     ▼
   Claude Code  ── skill 이 두 파일 읽는 법을 알고 있어 즉시 작업 가능


 3.2 캡쳐 모듈 (node-screenshots)
 --------------------------------------------------------------------------------
   - 영역 캡쳐 (/capture-region)
       모니터 전체 캡쳐 → 브라우저로 보냄 → 사용자가 캔버스에서 사각형 크롭.
       (전통적 "찍기 전 드래그" = 네이티브 투명 오버레이 필요 → 더 무거움, MVP 제외)
   - 지정 윈도우 캡쳐 (/capture-window)
       Window.all() 로 목록(제목·앱명·좌표·id) → 브라우저 썸네일 picker 에서 선택
       → 해당 window.captureImage().
   - 멀티 모니터: Monitor.all() 순회. 어느 모니터인지 메타에 기록.


 3.3 주석/내보내기 (브라우저 측에서 PNG 굽기 — 서버 렌더 의존성 제거)
 --------------------------------------------------------------------------------
   - 브라우저 canvas 가 이미 [원본 이미지 + 번호 박스]를 렌더 중이므로,
     canvas.toBlob()/toDataURL() 로 annotated.png 를 브라우저가 직접 export.
   - 서버는 받은 PNG + JSON 을 그대로 저장 → node-canvas/sharp 같은 네이티브
     렌더 의존성 불필요 (가벼움 유지).
   - (대안) 서버측 렌더가 필요하면 sharp(SVG 오버레이 합성) 고려. 기본은 브라우저 export.


 3.4 트리거 설계 (전역 핫키 대체 — 3종)
 --------------------------------------------------------------------------------
   ① 슬래시 명령  [MVP 기본, 100% 플러그인, 핫키 불필요]
        /capture-region , /capture-window , /annotate <path>
        → 어차피 Claude Code 안에서 작업 중이므로 핫키가 사실상 불필요.

   ② OS 기본 캡쳐 핫키 + /annotate  [이미 존재하는 전역 핫키 활용]
        Win+Shift+S (mac ⌘⇧4) 로 스닙 → 클립보드 → /annotate
        → MCP 가 클립보드 이미지를 읽어 주석 페이지 오픈.
        ※ 알려진 거친 부분: Node 에서 "클립보드 이미지" 읽기는 per-OS 접착코드 필요
          (Win: PowerShell/.NET, mac: pngpaste/AppleScript, Linux: wl-paste/xclip).
          MVP 에선 ① 네이티브 캡쳐가 더 깔끔 → ② 는 편의 옵션으로 후순위.

   ③ 커스텀 전역 핫키  [옵션, 나중에 / 플러그인 밖 유일한 조각]
        작은 OS 헬퍼(트레이앱/AutoHotkey)가 전역 핫키 소유 → MCP/파일로 전달.
        Win: AutoHotkey 몇 줄, 크로스플랫폼: 작은 네이티브 트레이앱.

 5. 데이터 형식 (핸드오프 스펙)
================================================================================

 5.1 산출물 (프로젝트 내 co-located, 기본 출력 디렉터리 예: ./shots/)
 --------------------------------------------------------------------------------
   <name>.annotated.png   원본 + 번호 박스가 구워진 이미지 (시각 앵커)
   <name>.regions.json    범례: 좌표 + 라벨 + 메모 (사람 편집 가능, 영속 메모리)
   (옵션) <name>.png       주석 없는 원본 (필요 시)

 5.2 regions.json 스키마 (초안)
 --------------------------------------------------------------------------------
   {
     "schema": "imagetoolforllm/regions@1",
     "createdAt": "2026-06-23T11:00:00Z",     // ISO8601
     "source": "region | window | clipboard",
     "window": {                               // source=window 일 때만
       "title": "Login - MyApp",
       "app": "MyApp"
     },
     "image": {
       "annotated": "login.annotated.png",
       "original": "login.png",                // 없으면 null
       "width": 960,
       "height": 600
     },
     "bboxFormat": "xywh",                      // [x, y, w, h] (좌상단 기준, px)
     "regions": [
       { "id": 1, "label": "Menu button", "note": "좌측 상단 햄버거",
         "bbox": [12, 8, 44, 44], "color": "#e53935" },
       { "id": 2, "label": "Exit button", "note": "우측 상단 X",
         "bbox": [904, 8, 44, 44], "color": "#1e88e5" }
     ]
   }

   ※ bbox 규약은 한 가지로 못박는다 → "xywh" (x, y, width, height, px, 좌상단 원점).
     README 예시는 [x1,y1,x2,y2] 처럼 보였으나, 혼선 방지를 위해 본 설계는 xywh 채택.
     (스키마에 bboxFormat 필드를 둬서 명시 + 향후 변경 여지 확보)

 5.3 영속성 / 재오픈
 --------------------------------------------------------------------------------
   - /annotate <path/to/x.annotated.png> → 옆의 x.regions.json 을 로드해 박스 복원.
   - 사용자가 재편집 후 다시 저장하면 같은 파일 갱신. (= 안 까먹는 메모리 역할)

 5.4 skill 이 Claude 에게 가르치는 것 (SKILL.md 요지)
 --------------------------------------------------------------------------------
   - "*.annotated.png 옆에 동일 이름 *.regions.json 이 오면 둘 다 읽어라"
   - "이미지 속 번호 박스 N ↔ regions[].id N ↔ label/note 로 해석하라"
   - bbox 좌표 규약(xywh) 설명, 좌표보다 시각 앵커(번호)를 우선 신뢰하라는 가이드.

 6. 크로스플랫폼 지원 매트릭스
================================================================================

   기능              | Windows | macOS                | Linux X11 | Linux Wayland
   ------------------|---------|----------------------|-----------|----------------
   영역 캡쳐         |   ✅    | ✅ (화면기록 권한)   |    ✅     | 🟡 portal 경유
   지정 윈도우 캡쳐  |   ✅    | ✅ (화면기록 권한)   |    ✅     | 🔴 핸들캡쳐 막힘
   윈도우 목록 열거  |   ✅    | ✅                   |    ✅     | 🟡
   브라우저 캔버스   |   ✅    | ✅                   |    ✅     | ✅
   슬래시 트리거     |   ✅    | ✅                   |    ✅     | ✅
   커스텀 전역핫키③  |   ✅    | ✅ (접근성 권한)     |    ✅     | 🔴 제한

     - Linux  = X11 양호 / Wayland 가 메인 나중 과제 (특히 지정 윈도우 캡쳐 + 전역핫키)
